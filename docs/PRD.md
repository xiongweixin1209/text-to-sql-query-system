## 一、产品背景与问题定义

### 1.1 问题的起点

数据分析的需求几乎存在于每一个业务决策场景中。但在大多数团队里，业务人员提出一个分析问题，到最终拿到答案，中间要经过需求沟通、SQL 编写、数据导出、结果交付等多个环节，响应周期往往以天计算。

这条链路的核心矛盾不是技术问题，而是**分析能力的不对等**——有分析需求的人不会写 SQL，会写 SQL 的人又不能实时响应所有需求。

### 1.2 现有方案的不足

Text-to-SQL 是解决这个问题的主流方向，但已有的工具普遍有几个共同的局限：

一是只做了 SQL 生成，没有覆盖到结果解读这一层。用户拿到 SQL 执行结果，还是一张原始数据表，仍然需要自己理解数据的业务含义。

二是依赖云端大模型，企业的核心业务数据需要上传至第三方服务，在数据安全和合规层面存在实际障碍。

三是对复杂数仓环境的适配不足。真实的企业数据库往往有数十张表、多层分层结构，通用工具难以在这种环境下准确理解用户意图。

### 1.3 这个产品想解决什么

这个项目的目标是构建一条从"业务问题"到"数据结论"的完整本地化链路。用户的输入是自然语言描述的业务问题，输出是图表加文字的分析结论，SQL 只是中间的执行过程。

整个链路在本地运行，不依赖外部云服务，适合对数据安全有要求的企业场景。

## 二、用户画像与使用场景

### 2.1 目标用户

这个产品的核心用户不是数据工程师，而是**有分析需求但不具备 SQL 能力的业务人员**。

运营、产品经理、管理层是最典型的目标群体。他们清楚自己想知道什么，但无法自己动手取数。另一类用户是数据分析新手，比如刚入职的实习生或业务分析师，会提问题但 SQL 能力有限。数据工程师也是潜在用户，主要用于快速验证数据，减少重复性的取数工作。

### 2.2 典型使用场景

**即时数据查询**：运营人员想知道上个月哪个商品销售额最高，直接输入这句话，几秒内得到排名结果和业务解读，不需要等待数据团队响应。

**数仓结构探索**：新加入的分析师面对一个陌生的数据库，通过系统的分层展示和业务域识别，快速建立对数据结构的整体认知，而不需要逐张表阅读文档。

**模糊需求的起点**：用户提出"分析客户的购买行为"这样比较宽泛的需求，系统推荐最相关的数据表，用户一键跳转并开始查询，降低分析的启动成本。

## 三、V1.0 功能说明与决策逻辑

### 3.1 已实现的功能

| 功能模块 | 说明 |
|---|---|
| 自然语言转 SQL | 三层策略（Rule / Few-shot / Zero-shot），120 条示例库覆盖常见查询类型 |
| 智能表推荐 | 调用本地 LLM 根据查询语义推荐数据表，标注置信度与推荐理由 |
| 数仓分层识别 | 自动识别 DWD/DWS 层级，按业务域归类展示 |
| SQL 执行与结果展示 | 支持分页、列排序、筛选、导出 CSV |
| 数据可视化 | 规则驱动的多图表推荐（柱状/折线/饼图/散点），无 LLM 开销 |
| AI 业务解读 | 调用本地 LLM 对查询结果给出 1-3 句业务层面的总结 |
| SQL 优化与性能分析 | 静态优化建议 + EXPLAIN 执行计划展示 |
| 多数据源管理 | 支持 SQLite / MySQL / PostgreSQL |

### 3.2 关于功能边界的判断

V1.0 的功能范围是经过权衡后的主动选择，而不是能力的上限。背后有两个关键判断。

**第一，本地模型的能力边界决定了哪些事情现在不该做。**

V1.0 使用本地部署的 Qwen2.5-Coder 7B。这个模型在 SQL 生成这类结构化任务上表现稳定，但在需要多步推理和复杂判断的任务上存在明显局限。

以图表类型选择为例：让模型基于查询语义判断"该用柱状还是折线"，需要它综合数据分布、业务含义和用户意图，7B 参数量在这类任务上容易误判。因此 V1.0 把这类判断从 LLM 中剥离出去，改用确定性规则承担——把不稳定的部分留给云端模型（V2.0 方向），本地链路只做能稳定做对的事。

**第二，场景不确定时，过度设计是一种浪费。**

可视化是这个问题最典型的体现。不同业务场景需要的图表形态差异很大，如果要做通用可视化，依赖云端大模型语义判断成本最低，但 V1.0 选择本地链路；而本地 7B 模型在"判断该用什么图表"这类任务上稳定性不足。

为此，V1.0 采用**规则驱动的有限多图表策略**：后端 `/api/text2sql/recommend-chart` 端点基于查询意图关键词 + 列数据类型做轻量推断（时间序列→折线、占比/分布→饼图、双数值列→散点、其余→柱状），前端 `SmartChart` 组件在客户端二次校验数据形态后渲染。判断逻辑全部用确定性规则，不调用 LLM，规避了模型误判的不稳定性。

未支持的图表类型保持表格展示，不强行渲染。本质仍是：**在不确定性中找到最可靠的立足点，而不是用复杂度掩盖不确定性。**

## 四、V2.0 产品规划

### 4.1 核心方向：场景收敛

V1.0 验证了整条技术链路的可行性。V2.0 的核心转变是**场景收敛**——以飞书多维表格作为第一个真实落地场景，在确定的场景边界内把每个功能做深，而不是继续维持通用性。

场景确定之后，AI 的判断逻辑有了明确的输入输出边界，图表类型的选择、结果的呈现方式，都可以在这个具体场景下被清晰定义。与此同时，接入云端模型（火山方舟）之后，原本受本地模型制约的功能也具备了实现条件。

### 4.2 功能规划

**飞书多维表格数据接入**（P0）

通过飞书开放平台 API 将多维表格接入作为数据源，用户在飞书内完成从提问到结论的完整闭环，不需要切换工具。

**LLM 驱动的图表智能化**（P0）

V1.0 的图表选择基于规则引擎（关键词 + 列类型），覆盖率有限。V2.0 接入云端模型后，由 LLM 综合查询语义、列分布、业务上下文判断图表类型与配置（含双轴、堆叠、分面等更复杂形态），并能解释为什么选这种图。这是受本地 7B 模型能力制约才剥离出来的部分，在云端模型下回归 LLM 判断。

**多轮对话式分析**（P1）

支持基于上次查询结果的追问。例如用户先查询了销售额排名，再问"其中第一名的月度趋势是什么"，系统理解上下文并生成新的查询。

**主动数据洞察**（P2）

Agent 在完成查询后，主动识别数据中的异常值或值得关注的规律，不等用户追问，直接输出洞察建议。

**分析报告导出**（P2）

将查询、图表、AI 解读打包为飞书文档或 PDF 报告，支持直接分享给团队。

### 4.3 演进路径

```
V1.0（当前）
本地模型 · SQLite · 通用场景 · 单轮查询 · 条件图表
        ↓
V1.5（过渡）
接入云端模型 · 完善图表判断 · 基础多轮对话
        ↓
V2.0（目标）
飞书多维表格接入 · 智能图表 · 多轮对话 · 主动洞察 · 报告导出
```
